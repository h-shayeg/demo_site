A Practical Introduction to Graph Neural Networks for Chemists

Introduction

If you've worked with molecular data before, you've probably represented molecules as fixed-length vectors — Morgan fingerprints, computed descriptors, MACCS keys — and fed them into whatever model you were using that week. That approach works, up to a point. This post is about the point past which it stops working, and what graph neural networks do differently.

Why Molecules Are Graphs, Not Vectors

A molecule is not naturally a vector. It's a graph: atoms are nodes, bonds are edges, and the properties you care about — reactivity, solubility, toxicity — emerge from that structure, not from an arbitrary list of counts. Fingerprinting throws the graph away and keeps a lossy summary of it. Most of the time that's fine. Near the edges of your training distribution, it usually isn't.

The failure mode isn't that fingerprints are wrong. It's that they're a summary computed once, up front, with no way to ask a slightly different question later.

Message Passing, Intuitively

Graph neural networks keep the structure. The core idea, common to most GNN architectures, is message passing: each node starts with a feature vector (atom type, charge, hybridization), and in each layer, every node updates its representation by aggregating messages from its neighbors. Stack enough layers and a node's representation eventually incorporates information from atoms several bonds away — exactly the kind of long-range structural context a fingerprint can't capture.

The Math

Formally, a single message-passing layer updates each node's hidden state using its previous state and an aggregation over its neighbors:

h_v^(k+1) = UPDATE( h_v^(k), AGGREGATE({ h_u^(k) : u ∈ N(v) }) )

Different architectures mostly differ in how UPDATE and AGGREGATE are implemented — a simple sum for GCN, an attention-weighted combination for GAT, and so on. The details matter for performance, but the shape of the idea is the same across all of them.

A Minimal Implementation

Here's roughly the smallest useful GNN, using PyTorch Geometric's graph attention layer:

simple_gnn.py Python 
import torch
from torch_geometric.nn import GATConv

class SimpleGNN(torch.nn.Module):
    def __init__(self, in_dim, hidden_dim, out_dim):
        super().__init__()
        self.conv1 = GATConv(in_dim, hidden_dim)
        self.conv2 = GATConv(hidden_dim, out_dim)

    def forward(self, x, edge_index):
        x = self.conv1(x, edge_index).relu()
        return self.conv2(x, edge_index)

What Can Go Wrong

The failure mode that catches almost everyone the first time: a random train/test split lets your model implicitly memorize scaffolds it has already seen in slightly different form, inflating validation accuracy. A scaffold split — grouping molecules by core structure so the same scaffold never appears in both train and test — gives a much more honest picture, and it's usually a less flattering one.

Where to Go From Here

None of this makes GNNs a universal upgrade — for small datasets, a well-tuned fingerprint-based model is often still the right call. But once you have enough data and care about generalizing past your training scaffolds, working with the graph directly stops being a nice-to-have and starts being the difference between a model that works in the lab and one that only works on the benchmark. MolGraphNet is the library I built around exactly this idea, if you want to see it applied end to end.
