From Jupyter to Production: Deploying ML for Materials Science

Kai Rutherford·Jun 2, 2026·8 min read

Introduction

Every research group eventually ends up with a notebook that "works" — a script someone can run top to bottom to get an answer. The problem is that "works on my machine, sometimes" isn't the same as "works whenever someone on the team needs it." That gap is what this post is about, using the reaction-yield model behind the Neural Reaction Predictor as a running example.

The problem with notebooks

Notebooks are great for exploration and terrible for anything anyone else needs to depend on. Cell execution order, hidden state, and unpinned dependencies mean a notebook that ran perfectly in March can quietly fail in June. For a model the lab wanted to query before every reaction screen, that unreliability was a real cost, not a nuisance.

  • State could get out of sync with the code on screen
  • Dependencies drifted between environments
  • There was no way for a labmate to query the model without opening the notebook themselves

Building a minimal inference service

The fix wasn't complicated: wrap the trained model in a small HTTP service with one job — accept a molecule, return a predicted yield and a confidence interval. Keeping the service's responsibility narrow made it easy to reason about and easy to test.

The goal isn't to build a platform. It's to build the smallest thing that lets someone else press a button instead of opening your notebook.

Packaging the model

The trained weights, the preprocessing code, and the exact library versions all had to travel together. Pinning dependencies and containerizing the service meant the model that got tested was the model that got used — no "it worked when I trained it" surprises three weeks later.

Deploying and monitoring

Once the service was running, the last piece was visibility: logging every prediction alongside the eventual real-world result, so the model's accuracy over time was something we could actually see rather than assume. That log became the input for the active-learning loop described in the Neural Reaction Predictor project.

Conclusion

None of this required heavyweight infrastructure — just treating the model like a piece of software the lab depends on, rather than a personal script. That shift in mindset mattered more than any specific tool choice.
