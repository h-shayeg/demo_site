Notes on Reproducibility in Computational Chemistry Pipelines

By Kian Rostami·March 2026·7 min read

On this page

  • Introduction
  • What Actually Breaks
  • Pin Everything, Including the Data
  • A Minimal Reproducible Config
  • Conclusion

Introduction

Most computational chemistry results can't be rerun two years later. This isn't a controversial claim — ask anyone who has tried to revisit their own PhD code for a journal revision. Here's what breaks, and the small set of practices that actually fix it.

What Actually Breaks

It's rarely the model itself. Model architectures are usually well-documented, sitting in a paper's methods section. What breaks is everything around the model: a specific RDKit version that quietly changed its canonical SMILES output, a dataset file that got overwritten with a "cleaned" version and no history, a random seed that was never recorded, a preprocessing script that lived only in someone's Jupyter kernel and was never saved. None of these show up in a paper. All of them determine whether your numbers are reproducible.

The paper describes the model. It almost never describes the twelve small decisions that got the data into a shape the model could use.

Pin Everything, Including the Data

The fix is unglamorous: pin your environment — not just top-level packages, but the full resolved dependency tree — version your data with content hashes rather than filenames, and log the exact configuration used for every run, not just the final one you report. None of this requires new tooling; most of it is a lockfile, a Dockerfile, and discipline about not silently overwriting files.

A Minimal Reproducible Config

In practice, this comes down to two artifacts that should exist for every run that ends up in a figure or table:

run_manifest.json JSON 
{
  "config_hash": "a3f9e21",
  "data_hash": "sha256:9e21a3f4c8b1...",
  "seed": 42,
  "git_commit": "4b7c9d1",
  "dependencies": {
    "rdkit": "2024.03.5",
    "torch": "2.3.1",
    "torch-geometric": "2.5.3"
  }
}

A config file and a lockfile, both checked in alongside the code, turn "what exactly produced Figure 3" from an archaeology project into a lookup.

Conclusion

None of this is intellectually hard. It's just tedious enough that it gets skipped under deadline pressure, and by the time anyone asks for the exact pipeline that produced a result, the moment to have done it properly has already passed. ChemPipe exists mostly to make the tedious version the default, not something you have to remember to do.
